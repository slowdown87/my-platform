## 服务选择篇

### 1. Cloudflare 在国内打不开

**问题**：最初选择 Cloudflare 作为托管平台，但部署后国内无法访问。

**原因**：Cloudflare 的部分服务在中国大陆被屏蔽。

**解决方案**：❌ 放弃 Cloudflare，改用其他平台。

---

### 2. Google AI 在国内打不开

**问题**：选择 Google AI 作为 AI 对话服务，但 API 地址无法访问。

**原因**：Google 服务在中国大陆被屏蔽。

**解决方案**：❌ 放弃 Google AI，寻找国内可用的替代方案。

---

### 3. Groq 在国内打不开

**问题**：Google AI 不可用后，尝试换用 Groq，但同样无法访问。

**原因**：Groq 的服务在中国大陆也无法访问。

**解决方案**：❌ 放弃 Groq，继续寻找替代方案。

**最终选择**：✅ **阿里百炼**（国内直连，免费额度充足）

---

### 4. Vercel 部署的网站国内打不开

**问题**：使用 Vercel 部署网站成功，但国内无法打开部署的网址。

**原因**：Vercel 的服务器和域名在国内被屏蔽。

**解决方案**：❌ 放弃 Vercel，改用 Netlify。

---

### 5. Gitee Pages 不支持环境变量

**问题**：尝试用 Gitee Pages 作为国内部署方案，但发现不支持环境变量配置。

**原因**：Gitee Pages 是纯静态托管，不支持服务器端环境变量。

**解决方案**：❌ 放弃 Gitee Pages。

---

### 6. 最终方案：Netlify

| 平台 | 国内访问 | 环境变量 | 最终选择 |
|------|---------|---------|---------|
| Cloudflare | ❌ | ✅ | ❌ |
| Vercel | ❌ | ✅ | ❌ |
| Gitee Pages | ✅ | ❌ | ❌ |
| **Netlify** | **✅** | **✅** | **✅** |

## 开发踩坑篇

### 1. npm install 很慢

**问题**：运行 `npm install` 时下载速度非常慢，经常超时。

**原因**：npm 默认从国外服务器下载包。

**解决方案**：
```bash
npm config set registry https://registry.npmmirror.com
```

---

### 2. Supabase API Keys 找不到

**问题**：在 Supabase 控制台找不到 API Keys。

**原因**：在 General 页面找错了位置。

**正确位置**：**Settings → API Keys**（不是 General 页面）

---

### 3. 关闭邮箱验证找不到

**问题**：想关闭 Supabase 的邮箱验证，但找不到开关在哪里。

**原因**：在 Email 模板页面找错了。

**正确位置**：**Authentication → CONFIGURATION → Sign In / Providers → User Signups → Confirm email**

> ⚠️ **注意**：不是在 `Notifications → Email → Confirm sign up`（那是编辑邮件模板的，不是开关）

---

### 4. 登录页面用 Supabase 账号登不上

**问题**：在平台登录页面输入 Supabase 的账号密码，提示登录失败。

**原因**：混淆了两个"登录"概念。Supabase 的账号是用来管理后台的，平台的登录需要先**注册新账号**。

**解决方案**：在平台登录页面点"注册"标签，创建一个新账号。

---

### 5. TypeScript 报 "data declared but never read"

**问题**：VS Code 中 TypeScript 报错 `data declared but never read`。

**原因**：代码中声明了 `data` 变量但没有使用它。

**解决方案**：删除未使用的变量。例如：
```typescript
// ❌ 错误
const { data, error } = await supabase.auth.signUp({...})

// ✅ 正确
const { error } = await supabase.auth.signUp({...})
```

---

### 6. 修改代码后本地登录报 "fail to fetch"

**问题**：之前登录正常，修改代码后突然报 "fail to fetch" 错误。

**原因**：`supabase.ts` 文件里的 URL 或密钥被改错了。

**解决方案**：重新从 Supabase 控制台（Settings → API Keys）复制正确的 URL 和 anon public 密钥，粘贴到 `supabase.ts` 中。

---

### 7. 浏览器白屏报 "supabaseUrl is required"

**问题**：浏览器打开后白屏，F12 控制台报错 `supabaseUrl is required`。

**原因**：`.env` 文件中的环境变量没有被 Vite 正确加载。

**排查步骤**：
1. 确认 `.env` 文件在项目**根目录**（`D:\my-platform\.env`）
2. 确认文件名以 `.` 开头
3. 确认等号两边**没有空格**：
   ```
   # ❌ 错误（有空格）
   VITE_SUPABASE_URL = https://...
   
   # ✅ 正确（无空格）
   VITE_SUPABASE_URL=https://...
   ```
4. 修改 `.env` 后需要**重启开发服务器**（Ctrl+C 然后 npm run dev）

---

### 8. 底部导航显示成垂直列表

**问题**：底部导航栏的图标和文字变成了垂直排列，而不是水平排列。

**原因**：`App.css` 中缺少 `.bottom-nav` 和 `.nav-item` 的 CSS 样式。

**解决方案**：更新 `App.css`，添加底部导航栏的样式：
```css
.bottom-nav {
  display: flex;
  justify-content: space-around;
  padding: 10px 0;
  background: white;
  border-top: 1px solid #eee;
}

.nav-item {
  display: flex;
  flex-direction: column;
  align-items: center;
  cursor: pointer;
  color: #666;
}
```

---

### 9. 环境变量有的有 VITE_ 前缀有的没有

**问题**：`.env` 文件里有的变量有 `VITE_` 前缀，有的没有，搞不清楚为什么。

**原因**：Vite 框架只会把 `VITE_` 开头的变量暴露给前端代码。

| 使用位置 | 环境变量名 | 访问方式 |
|---------|-----------|---------|
| 前端代码（浏览器） | `VITE_SUPABASE_URL` | `import.meta.env.VITE_SUPABASE_URL` |
| 服务器端（Netlify Functions） | `BAILIAN_API_KEY` | `process.env.BAILIAN_API_KEY` |

**安全考虑**：
- Supabase 的 `anon key` 是公开的，放前端没问题
- 百炼 API Key 是私密密钥，**不能暴露给前端**，只能在服务器端使用

---

### 10. qwen2.5-omni-7b 无法联网搜索

**问题**：添加了 `enable_search: true` 参数，但 AI 仍然无法获取最新信息，回复说知识截止到2021年。

**原因**：`qwen2.5-omni-7b` 模型**不支持** `enable_search` 联网搜索功能。

**解决方案**：换成 `qwen-turbo` 模型（支持联网搜索且免费）。

**支持联网搜索的免费模型**：
- ✅ qwen-turbo
- ✅ qwen-plus
- ✅ qwen-flash
- ❌ qwen2.5-omni-7b（不支持）

---

### 11. 百炼联网搜索收费

**问题**：使用百炼的 `enable_search: true` 后，阿里云账户显示欠费。

**原因**：百炼的联网搜索功能**不在免费额度内**，每次调用收费 ¥0.003。

**费用详情**：
- 联网搜索：¥0.003/次
- 普通对话（不开联网搜索）：免费

**解决方案**：使用 **Tavily** 代替百炼自带的联网搜索。
- Tavily：每月 1000 次免费，不需要绑卡
- 注册地址：https://tavily.com
- 实现方式：先用 Tavily 搜索，再把结果发给百炼 AI 总结

详见：[step9-免费联网搜索.md](./step9-免费联网搜索.md)

## 部署踩坑篇

### 1. git push 报 "Repository not found"

**问题**：执行 `git push origin main` 时报错 `fatal: remote origin already exists` 或 `Repository not found`。

**原因**：GitHub 上还没有创建对应的仓库。

**解决方案**：
1. 打开 https://github.com/new
2. 仓库名填 `my-platform`
3. 选择 Public 或 Private
4. 点击 **Create repository**
5. 然后再 `git push`

---

### 2. git push 要求密码但输不对

**问题**：`git push` 时要求输入用户名和密码，但密码总是不对。

**原因**：GitHub 从 2021 年开始不再支持密码认证。

**解决方案**：使用 **Personal Access Token（PAT）** 代替密码：
1. 打开 GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
2. 点击 **Generate new token**
3. 勾选 `repo` 权限
4. 生成后复制 Token
5. `git push` 时：
   - Username：你的 GitHub 用户名
   - Password：粘贴刚才的 Token

---

### 3. Netlify 报 "dist does not exist"

**问题**：Netlify 部署时报错 `Deploy directory 'dist' does not exist`。

**原因**：`vite.config.ts` 中的 `outDir` 被改成了 `docs`。

**解决方案**：删除 `vite.config.ts` 中的 `outDir` 配置，使用默认的 `dist`：
```typescript
// ❌ 错误
export default defineConfig({
  plugins: [react()],
  build: {
    outDir: 'docs'  // 删除这行
  }
})

// ✅ 正确
export default defineConfig({
  plugins: [react()],
})
```

---

### 4. Netlify 环境变量在 Team settings 找不到

**问题**：在 Netlify 的 Team settings 里找环境变量，找不到添加的地方。

**原因**：找错了位置。环境变量在**站点配置**里，不是团队设置里。

**正确位置**：
1. 选择您的站点
2. 左侧菜单 → **Site configuration**
3. 点击 **Environment variables**
4. 点击 **Add a variable**

---

### 5. NetlifyUserError: Function returned unsupported value

**问题**：Netlify Functions 返回错误 `Function returned an unsupported value. Accepted types are 'Response' or 'undefined'`。

**原因**：Netlify Functions v2 不再支持旧的 Lambda 格式返回值。

**解决方案**：使用 `new Response()` 格式：
```javascript
// ❌ 错误（旧版格式）
export default async function handler(event) {
  return { statusCode: 200, body: JSON.stringify(data) }
}

// ✅ 正确（v2 格式）
export default async (request, context) => {
  return new Response(JSON.stringify(data), {
    status: 200,
    headers: { 'Content-Type': 'application/json' }
  })
}
```

---

### 6. Netlify Functions 报 "Method not allowed"

**问题**：函数能调用，但返回 `Method not allowed`。

**原因**：函数参数格式错误，使用了旧的 `handler(event)` 格式。

**解决方案**：使用标准的 `Request` 对象参数：
```javascript
// ❌ 错误
export default async function handler(event) {
  if (event.httpMethod !== 'POST') { ... }
  const { messages } = JSON.parse(event.body)
}

// ✅ 正确
export default async (request, context) => {
  if (request.method !== 'POST') { ... }
  const { messages } = await request.json()
}
```

---

### 7. GitHub Pages 空白页

**问题**：部署到 GitHub Pages 后打开是空白页。

**原因**：`vite.config.ts` 的 `base` 路径配置不正确。GitHub Pages 需要子路径 `/my-platform/`，而 Netlify 需要根路径 `/`。

**解决方案**：用环境变量动态控制 `base` 路径：
```typescript
// vite.config.ts
import { defineConfig, loadEnv } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')
  return {
    plugins: [react()],
    base: env.VITE_BASE_URL || '/my-platform/',
  }
})
```

```toml
# netlify.toml
[build.environment]
  VITE_BASE_URL = "/"
```

```bash
# 部署 GitHub Pages 时临时覆盖
set VITE_BASE_URL=/my-platform/ && npm run build && npx gh-pages -d dist
```

### 8. GitHub 不稳定

**问题**：`git push` 经常超时或连接失败。

**原因**：国内网络访问 GitHub 不稳定。

**解决方案**：代码同时推送到 Gitee 作为备份。

## 重要经验总结

1. **国内环境要做适配**：很多国外服务（Cloudflare、Google、Groq、Vercel）在国内无法使用，需要提前准备替代方案
2. **环境变量要提前配**：部署前一定要先配好环境变量，否则部署出来网站打不开
3. **.env 文件不会上传**：Git 会忽略 .env 文件，所以部署平台需要单独配置密钥
4. **Personal Access Token**：GitHub 已经不支持密码推送代码，需要生成 Token
5. **Public vs Private**：GitHub 仓库选 Public 或 Private 都可以，.env 不会被上传
6. **anon public 密钥是安全的**：它本来就是公开的，写死在代码里也没关系
7. **Netlify Functions v2**：必须用 `(request, context)` 参数和 `new Response()` 返回格式
8. **模型要选对**：不是所有模型都支持所有功能，比如联网搜索需要特定的模型
9. **双部署路径问题**：用 `VITE_BASE_URL` 环境变量区分 GitHub Pages 和 Netlify 的 base 路径
